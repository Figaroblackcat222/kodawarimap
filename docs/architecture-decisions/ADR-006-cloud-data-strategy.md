# ADR-006: クラウドバックアップ／公開のデータ戦略

## 状況
将来のクラウドバックアップとWeb公開機能で、何をクラウドに置くかでコストが3桁変わり、核心価値「プライバシー」とも衝突しうる。MVPには影響しないが今のうちに方針を確定する。

## 前提（コスト分析）
- R2ストレージ超過は$0.015/GB-月と激安、egressは無料。「無料枠の天井」は重要指標ではない
- 日本全域PMTilesベース地図（5〜10GB）だけで無料枠10GBをほぼ消費するが、超過分は実害月数十円
- コストを支配するのは「写真をクラウドに上げるか」。フル解像度同期は1ユーザー約900MB/年、メタ＋サムネは約45MB/年、メタのみは約0.6MB/年

## 決定
1. **クラウドバックアップ = メタデータ＋サムネイルのみ。原本写真は端末に残す**
2. **Web公開 = バックアップとは別ストア。本人が明示選択したスポットのみ、Web最適化画像をアップロード（プロ版課金）**
3. **画像最適化 = クライアント側でアップ前にリサイズ**（原本が端末から出ない／プライバシー相乗）。運用簡素化したい場合は Cloudflare Images を代替候補
4. **公開ストア = Cloudflare R2**（egress無料・タイル/Backupと同一ベンダ集約）
5. **フル解像度クラウド保管 = 将来の上位課金プラン**

## 理由
- バックアップと公開は目的が違う（災害復旧 vs 意図的発信）。公開をバックアップストアに依存させない
- コストが収益と一致（公開はopt-in＆一部＆プロ版課金、暴走しない）
- 原本が自動でクラウドに出ない → 「公開範囲を100%コントロール」の要件と整合
- Clean Architecture的に `BackupRepository` と `PublishRepository` は別port、`PublishSelectedMap` は独立UseCase、Domainは無傷

## 既知のトレードオフ
端末紛失かつバックアップがサムネのみの場合、フル画質での再公開はできない。これは欠陥ではなく「フル解像度クラウド保管庫＝上位課金プラン」への自然な動線。

## 将来依存
公開（プロ版）は「誰がどのマップを公開したか」の認証＋小さなDBが別途必要。将来のバックエンド決定に持ち越し（[ADR-002](ADR-002-no-backend-mvp.md)）。
